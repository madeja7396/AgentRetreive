# 実装でしてはいけないこと（Do Not）

更新日: 2026-02-26

## 目的

- 品質劣化、再現性崩壊、論文証跡欠損を防ぐ
- 実装判断のブレを減らす

## 1. 契約（Schema/SSOT）違反

- `docs/schemas/` を更新せずに入出力仕様を変更してはいけない
- 予約キー/予約prefix（`docs/NAMESPACE_RESERVATIONS.md`）を再定義してはいけない
- 破壊的変更を v1 に上書きしてはいけない（新 version を追加する）
- `x_` 接頭辞なしに未定義拡張キーを追加してはいけない

## 2. 再現性を壊す実装

- 実験で未固定バージョン依存（latest 依存）を導入してはいけない
- ランダム性を使う処理で seed を記録せずに実行してはいけない
- 実験結果だけを保存し、設定/commit/環境情報を保存しない運用をしてはいけない
- 失敗 run を削除してはいけない（原因分析不能になる）

## 3. データ管理違反

- `artifacts/datasets/raw/` の生データを上書きしてはいけない
- dataset manifest なしでデータを評価に使ってはいけない
- ライセンス未確認データをベンチ/公開成果物に混ぜてはいけない
- 追跡不能な手動加工データを論文根拠に使ってはいけない

## 4. 出力・性能設計の逸脱

- `max_bytes` 等の budget 制約を無視して結果を返してはいけない
- 決定性要件（同一入力で同一出力）を壊す並び順を導入してはいけない
- スコア仕様（0-1000 整数）を無断で変更してはいけない
- LLM 向け出力に不要な冗長情報（巨大パス反復など）を増やしてはいけない

## 5. 評価・論文運用の禁止事項

- 手作業で図表値を編集してはいけない（必ずスクリプト生成）
- 主張に対応する実験 ID がないまま論文化してはいけない
- ベースライン条件を明示せず比較結果を提示してはいけない
- 再現試験未実施の結果を「確定」と扱ってはいけない

## 6. Git/開発運用の禁止事項

- `main` に未検証の変更を直接 push してはいけない
- CI 失敗状態でマージ相当の運用をしてはいけない
- ローカル生成物（`artifacts/`, `dist/`, 仮想環境）を追跡対象にしてはいけない
- 履歴を壊す強制操作（`reset --hard` 等）を合意なしに実行してはいけない

## 7. 常駐デーモン運用の禁止事項

- `agentd` に任意シェル実行タスクを追加してはいけない
- schema 未登録の task type を投入してはいけない
- `dead` タスクを原因分析なしに再投入してはいけない
- `in_progress` ファイルを手動編集してはいけない

## 8. 実装前チェック（簡易）

- [ ] 変更対象仕様は `docs/` 側で先に定義したか
- [ ] `bash scripts/ci/run_contract_harness.sh` は通ったか
- [ ] 実験証跡（run_id / dataset_id）を残せる設計か
- [ ] 破壊的変更時の version 戦略を決めたか
