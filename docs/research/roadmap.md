# AgentRetrieve 改善ロードマップ

**現状**: v2.0 実験完了（2026-02-26）  
**基盤**: 7リポジトリ71.4% recall達成、レイテンシ課題残存

---

## Phase 1: 緊急対応（レイテンシ改善）

**目標**: 大規模コーパス（curl: 1,849ファイル）でレイテンシ < 1秒

### 1.1 Posting List キャッシュ
- **内容**: 頻出語のposting listをメモリ上にキャッシュ
- **期待効果**: 2-5x 高速化
- **工数**: 小（1-2日）
- **実装案**: LRUキャッシュ、頻出語Top 100を保持

### 1.2 並列検索実装
- **内容**: 複数クエリ語のposting list走査を並列化
- **期待効果**: 2-3x 高速化（CPUコア数依存）
- **工数**: 小（2-3日）
- **実装案**: ThreadPoolExecutorによる並列スコア計算

### 1.3 Posting List 圧縮
- **内容**: 可変長符号化（Variable Byte Encoding）でインデックス圧縮
- **期待効果**: メモリ使用量50%削減、キャッシュ効率向上
- **工数**: 中（3-5日）
- **参考**: Luceneの圧縮アルゴリズム

---

## Phase 2: 精度改善（Recall/MRR向上）

**目標**: Overall Recall 80%+、MRR 0.5+

### 2.1 クエリ拡張（Query Expansion）
- **内容**: 類義語・派生形の自動展開
- **期待効果**: Recall +10-15%
- **工数**: 中（1週間）
- **実装案**:
  - 単語ベクトル（Word2Vec/FastText）による類似語検索
  - 編集距離によるスペル修正
  - コード固有の略語辞書（http→hypertext transfer protocol）

### 2.2 文脈スコアリング
- **内容**: ファイルパス、隣接コードを考慮したスコア調整
- **期待効果**: MRR +20%
- **工数**: 中（1週間）
- **実装案**:
  - ファイルパスとのマッチでボーナス（`parser/arg_parser.c` に「parse」クエリ）
  - 同じファイル内の他のシンボルとの共起性

### 2.3 型情報・スコープの統合
- **内容**: 静的解析による型・スコープ情報のインデックス追加
- **期待効果**: Recall +15%、特にsymbol_usageタスク
- **工数**: 大（2-3週間）
- **実装案**:
  - tree-sitterによる軽量パース
  - 型階層（継承関係）のインデックス化
  - 呼び出しグラフの構築

---

## Phase 3: 高度化（学習型アプローチ）

**目標**: MRR 0.7+、Hardタスク対応

### 3.1 学習型ランキング（LTR）
- **内容**: 特徴量ベースの再ランキング
- **期待効果**: MRR +30%
- **工数**: 大（2-3週間）
- **実装案**:
  - 特徴量: BM25スコア、ファイルサイズ、編集距離、型一致度
  - 軽量モデル: LambdaMARTまたはニューラルネット（2-3層）
  - 訓練データ: クリックログまたはタスクセットから自動生成

### 3.2 セマンティック検索（埋め込み）
- **内容**: コード・クエリをベクトル化し、コサイン類似度で検索
- **期待効果**: Hardタスク Recall +20%
- **工数**: 大（3-4週間）
- **実装案**:
  - 事前学習モデル: CodeBERT/CodeT5またはOpenAI Embeddings
  - ハイブリッド検索: BM25（キーワード）+ ベクトル（意味）の組み合わせ
  - 近似最近傍探索（FAISS/Annoy）

### 3.3 対話的検索（フィードバックループ）
- **内容**: ユーザーのクリック・フィードバックから学習
- **期待効果**: 個人化、長期的な精度向上
- **工数**: 大（1ヶ月）
- **実装案**:
  - Relevance Feedback: 正例を使ったクエリ拡張
  - セッション履歴の活用

---

## Phase 4: スケーラビリティ（大規模対応）

**目標**: 50,000+ファイル（Linuxカーネル規模）での実用化

### 4.1 分散インデックス
- **内容**: シャーディングによるインデックス分散
- **期待効果**: 線形スケールアウト
- **工数**: 大（1-2ヶ月）
- **実装案**:
  - ファイルパスベースのシャード分割
  - マージ結果の集約

### 4.2 インクリメンタルインデックス
- **内容**: 差分更新によるインデックス構築時間短縮
- **期待効果**: 大規模コーパスの日次インデックス更新を現実的に
- **工数**: 中（1-2週間）

### 4.3 階層的検索
- **内容**: モジュール→ファイル→関数の階層的絞り込み
- **期待効果**: 検索空間の削減、レイテンシ改善
- **工数**: 中（2週間）
