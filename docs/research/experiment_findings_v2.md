| fmt | C++ | 67 | **80.0%** | 0.300 | 63.9ms |

## 2. 詳細な知見

### 2.1 スケーラビリティの課題（重大）

```
ファイル数 vs レイテンシ:
    31 files   →   16ms  (fd)
    67 files   →   64ms  (fmt)
   253 files   →  373ms  (pytest)
   824 files   → 4,737ms (cli)
 1,849 files   → 15,893ms (curl) ← 実用不可
```

**知見**:
- レイテンシはファイル数に対して非線形（おそらく O(n log n) 以上）に増加
- 1,000ファイル以上のコーパスでは検索に15秒以上かかり、実用的でない
- ボトルネックはposting listの走査とBM25スコア計算
- インデックスサイズも問題：curlは25MB、メモリ圧迫の懸念

**根本原因**:
- 各クエリ語に対して全posting listを走査
- スコアリング時の逐次計算
- キャッシュ機構の欠如

### 2.2 難易度別性能の傾向

| 難易度 | Recall | タスク数 | 特徴 |
|--------|--------|---------|------|
| Easy | **85.7%** | 14 | シンプルなシンボル定義検索 |
| Medium | **64.3%** | 14 | 複合条件、使用箇所検索 |
| Hard | **57.1%** | 7 | 文脈依存、複雑な推論必要 |

**知見**:
- Easy→Mediumで**21ポイント**のrecall低下
- Medium→Hardでさらに**7ポイント**低下
- Hardタスクは4/7成功、3/7失敗（43%失敗率）

**失敗パターン分析**:
1. **曖昧なクエリ**: 複数の候補にマッチし、正解が低ランク
2. **文脈依存**: ファイルパスや隣接コードの文脈を考慮できない
3. **間接的参照**: マクロ、エイリアス、間接関数呼び出しの追跡不可

### 2.3 タスクタイプ別性能

| タスクタイプ | Recall | タスク数 | コメント |
|-------------|--------|---------|---------|
| symbol_definition | **85.7%** | 14 | 最も得意。識別子→定義の直接マッピング |
| symbol_usage | **未計測** | - | 定義検索に比べ困難（スコープ解析必要） |
| pattern_search | **未計測** | - | 複雑なパターンは正規表現に劣る可能性 |
| cross_reference | **未計測** | - | 呼び出しグラフ構築が必要 |

**知見**:
- symbol_definitionは安定して高い性能
- 他のタスクタイプについてはデータ不足だが、推定で50-60%程度と予想

### 2.4 リポジトリ特性の影響

**高性能リポジトリ（fd, fmt）の特徴**:
- 規模が小さい（<100ファイル）
- 命名規則が明確（camelCase/snake_case一貫性）
- シンボル名が一意で衝突が少ない

**低性能リポジトリ（curl, cli）の特徴**:
- 大規模（>500ファイル）
- 一般的な関数名（`init`, `parse`, `process`）の衝突
- 複雑なディレクトリ構造
- 多言語混在（C + Python + Perlなど）

**知見**:
- 本手法は「小規模・整備されたコーパス」に強い
- 「大規模・複雑なコードベース」では精度・速度両方で課題

### 2.5 パラメータチューニングの知見

**グリッドサーチ結果（300設定×複数リポジトリ）**:

| パラメータ | 最適値の範囲 | 影響度 | 考察 |
|-----------|-------------|--------|------|
| k1 | 0.8 - 1.5 | 中 | 小さいほど厳密マッチ重視。大きいほど頻度重視 |
| b | 0.3 - 1.0 | 低 | 文書長正規化。リポジトリ特性による差が小さい |
| **min_match_ratio** | **0.0 - 0.5** | **高** | **最重要パラメータ** |
| max_terms | 2 - 3 | 中 | 3以上はノイズ増加のみ |

**最重要発見: min_match=0.5の効果**:
- min_match=0.0（厳密AND）: Recall 30-50%
- min_match=0.5（部分緩和）: Recall 60-80% (**+20-30ポイント改善**)
- min_match=0.75以上: ノイズ増加、Precision低下

**クエリ緩和戦略の効果**:
```
クエリ「find file recursive」
- strict (min_match=0.0): [find, file, recursive] すべて必須 → Recall低
- relaxed (min_match=0.5): [find] must + [file, recursive] should → Recall高
```

### 2.6 ランキング品質（MRR）の課題

**MRR分布**:
- ripgrep: 0.600（最良）
- fd: 0.550
- fzf, pytest: 0.333
- curl: 0.158（最悪）
- 全体平均: 0.346

**知見**:
- MRR 0.346 = 平均して3位以内に正解がない
- BM25スコアはTF-IDF的な関連度であり「コードの意味」を捉えていない
- 大規模コーパスではスコアの差が小さく、ランキングが不安定

**改善余地**:
- 型情報、シグネチャ、スコープを考慮したスコアリングが必要
- 学習型ランキング（LTR）の導入でMRR 0.5以上を目標

### 2.7 トークナイゼーションの効果と限界

**採用戦略**: camelCase/snake_case分割 + 小文字化

```
Input: "parseArguments"
Output: ["parse", "arguments"]

Input: "handle_http_request"
Output: ["handle", "http", "request"]
```

**効果**:
- 部分一致検索が可能に（「parse」で「parseArguments」にヒット）
- 単語境界の明示化

**限界**:
- 一般的な略語への対応不可（「http」「url」などは分割されない）
- 数字の扱いが不明確（「base64」→ ["base", "64"]）
- 文脈を考慮しないため多義語問題が残存

### 2.8 インデックス構造の分析

| リポジトリ | インデックスサイズ | 語彙数 | 文書数 |
|-----------|------------------|--------|--------|
| fd | 0.7MB | ~2,000 | 31 |
| curl | 25MB | ~45,000 | 1,849 |
| cli | 12MB | ~35,000 | 824 |

**知見**:
- インデックスサイズはおおよそ文書数に比例
- 語彙の希少性（rare terms）が検索性能に大きく影響
- 一般的な単語（"the", "and"等）はフィルタリング済み

## 3. 比較評価（git grepとの比較）

### 3.1 既存の比較結果（fdリポジトリ、5タスク）

| ツール | Recall | レイテンシ | 出力サイズ |
|--------|--------|-----------|-----------|
| AgentRetrieve | **100%** | **29ms** | 10KB |
| git grep | 60% | 305ms | 100KB+ |

**知見**:
- AgentRetrieveは**9倍高速**、**1.7倍高いRecall**
- 出力サイズは**1/10**（ノイズ削減）
- 小規模コーパスでは既存ツールを上回る

### 3.2 ripgrep比較の問題

**予期せぬ課題**: ripgrepが特定パターンでタイムアウト

- PCRE2機能がビルドに含まれていない（features:-pcre2）
- リポジトリ全体検索で無限ループに近い挙動
- ファイル単位検索は正常動作

**教訓**:
- 外部ツールとの公平な比較は困難（設定・環境依存）
- 比較実験には-F（固定文字列）フラグとファイル種別フィルタが必要

## 4. 限界と今後の課題

### 4.1 現在の限界

1. **セマンティック理解の欠如**
   - 型システムの理解なし
   - 制御フロー・データフローの追跡不可
   - コメントやドキュメントの活用なし

2. **大規模コーパス対応の非現実性**
   - 1,000ファイル超で15秒以上のレイテンシ
   - Linuxカーネル規模（50,000+ファイル）では完全に非実用

3. **クエリ表現の貧弱さ**
   - 自然言語クエリをDSLに変換する仕組みなし
   - 対話的な絞り込み（フィードバックループ）なし
   - 類義語・派生形の自動展開なし

4. **文脈情報の未活用**
   - ファイルパスの意味（階層構造）を無視
   - 最近の編集履歴を考慮しない
   - プロジェクト固有の規約を学習しない

### 4.2 改善優先度マトリクス

| 優先度 | 項目 | 期待効果 | 工数見積 |

## 5. 実用上の推奨事項

### 5.1 適用可能なユースケース

**推奨**:
- 小規模〜中規模プロジェクト（<500ファイル）
- 命名規則が明確なコードベース
- シンボル定義検索が中心のユースケース
- リアルタイム検索が必要なIDE統合

**非推奨**:
- 大規模プロジェクト（>2,000ファイル）
- 複雑なクロスリファレンス解析
- 自然言語での意図検索
- レガシーコード（一貫性のない命名）

### 5.2 パラメータ設定ガイドライン

```yaml
# 小規模プロジェクト（<100ファイル）
k1: 0.8
b: 1.0
min_match_ratio: 0.0  # 厳密マッチで十分
max_terms: 3

# 中規模プロジェクト（100-500ファイル）
k1: 1.2
b: 0.75
min_match_ratio: 0.5  # 緩和が有効
max_terms: 3

# 大規模プロジェクト（>500ファイル）
k1: 1.5
b: 0.5
min_match_ratio: 0.5
max_terms: 2  # ノイズ削減
# 注意: レイテンシ問題を先に解決すること
```

## 6. 結論

AgentRetrieveは**小規模コードベース向けの軽量コード検索エンジン**として機能する。最適化されたパラメータ（min_match=0.5が鍵）により、71%のrecallを達成。

**主な成果**:
- 300パラメータのグリッドサーチで最適設定を特定
- min_match緩和によりrecallを20-30%向上
- 小規模コーパスでgit grepを上回る性能

**重大な課題**:
- 大規模コーパスでのレイテンシ悪化（15秒以上）
- 難易度の高いタスクでの性能劣化（57% recall）
- ランキング品質の低さ（MRR 0.346）

**今後の方向性**:
- インデックス構造の最適化（キャッシュ、圧縮）
- セマンティック情報の統合（型、スコープ）
- 学習型ランキングの導入
